That is why I have always been a huge fan of the Pocket Reference series by O'Reilly Press. Who has time for 400 page tome on the latest big thing? If you understand a technology's core concepts and just want to get to the meat of the topic, you really cannot beat a good Pocket Reference. They are also about a fourth to a fifth of the cost of the bigger programming books, which helps if you are cheap frugal.

Since its announcement about a year ago, I have been wanting to play around with Swift, Apple's new development language for MacOS and iOS. Last month I discovered that O'Reilly has a Swift Pocket Reference. Amazon.com sells it for less than $13, and it's only about 140 pages long. I can get through that in a weekend. So I did.

The book doesn't waste much time discussing the history of software development, or what prompted Apple to create Swift. Instead, as all good Pocket References do, it dives right into content. If you have never seen or used a C-style language before, this is not a book to learn it. The book also doesn't go into any iOS or MacOS specific features.

However, if you are already familiar with Mac or iOS development, or Java or C#, then it won't take you long to grasp the language. The book wastes no time, getting to the language syntax and constructs -- built-in datatypes, variables and constants, operators, etc -- by page 14. It spends a decent amount of time on strings, arrays, and dictionaries.

Coming from a C# and .NET background, I appreciate a lot of the modern features that Swift provides. It is designed as an object-oriented language, but like C#, also has some functional goodness thrown in for good measure.

Strongly Typed with Inference

Swift is a strongly-typed language, just like C# and Java. However, it also supports type inference, just as C# does. If you create a variable with the statement var x = 0, x is inferred to be an int. Likewise, let name = "mike" allows the compiler to infer that name is a string. Don't be fooled. The types are fixed. You can't later assign "mike" to x or 42 to name.

Inheritance

Swift offers single-inheritance from a base class, which tends to result in narrow, but deep, class structures. Swift supports the inheritance of what Apple calls Protocols, but Java and C# developers would understand as Interfaces. Basically, a definition of a class properties and methods without an implementation. Different term. Same name. A class may only derive from one super class, but may implement as many protocols as the developer wishes.

Properties

Swift has properties, just like C#, but with a twist. You can create observable properties, so that your code can take action when the value changes. This seems familiar to the .NET IPropertyChange interface, but built-into the language, which seems cleaner.

Generics

Swift supports generics, so that you can avoid code duplication where multiple functions whose implementations differ only in the types they operate upon. Using generics, you can write a single function that can operate on any (or a limited set) of data types. A common example of this is a stack. Regardless of the types of objects you place on the stack, the behavior and implementation of the stack doesn't change. In the past, we could simply create a stack of "object" or another base class. However, then there is nothing to prevent you from pushing both a BankAccount and a PlayingCard onto the same stack. With generics, you can create a Stack of BankAccounts, or a Queue of Playing Cards, for example. Each would continue to be strongly-typed, and only accept and return values of that particular type.

Closures

Closures in Swift look like anonymous functions in JavaScript, or Lambda expressions in Java and C#. They are used to encapsulate and capture functionality and data, yet keep the inner details hidden from any calling code. Closures are often used to pass small bits of functionality into a larger function. 

One of the simpler examples of a closure is to pass a simple conditional to a sort function, as shown here: 
  • let reversed = sorted(names, { s1, s2 in s1 > s2 } )
In that example, given an array of strings in the variable names, we call the sorted function, passing in the names array, and a closure. The closure itself is the second parameter, including the curly braces. It indicates an anonymous function that takes two parameters, and returns a boolean defined by s1 > s2. Notice the type inference going on: the parameters types s1 and s2, and the boolean return type.

For reference, the similar function in C# would be:

  • var reversed = names.sort((s1, s2) => s1 > s2; );

Extensions

The "O" in the SOLID principle stands for Open/Closed. A type should be open for extension, but closed to modification. Extensions (Extension methods in C#) provide the ability to decorate an existing class with new functionality, without modifying the implementation of the class itself.

Conclusion

Overall, I believe this language is a good move for Apple. Anyone who didn't grow up thinking in Objective C probably has a hard time with it. But if you spent the past 10-15 years with Java or C#, as I did, Swift has a very natural feel to it. If you want to know what the language is all about, grab a copy of this book, the latest version of Xcode, and a spare day or two this weekend. You'll probably be glad you did.
